= Filling Hibernate's Gaps with Hypersistence Utils
Vlad Mihalcea
:source-language: java
:sourcedir: {examplesdir}

Hibernate maps the SQL standard well. Your database is not the SQL standard.

Somewhere between the two sits every column type you actually wanted to use: a
PostgreSQL `jsonb` document, a `text[]`, an `inet` address, a `tsrange`, a
`hstore`. And somewhere alongside them sit the everyday operations JPA never
quite got right: updating a detached entity without reading it first, retrying a
transaction that failed for a reason that will not repeat, finding out which
line of your code fired that query.

https://github.com/vladmihalcea/hypersistence-utils[Hypersistence Utils] is the
library that fills those gaps. This chapter walks through the parts of it worth
knowing, one working project at a time. Every listing you are about to read is
included straight from a project in this book's repository at build time — none
of it is retyped, and none of it can drift from the code that is actually
tested.

== Adding the library

Hypersistence Utils ships one artifact per Hibernate line, and you must match
yours. This book uses Hibernate 7.3, so the artifact is
`hypersistence-utils-hibernate-73`:

[source,xml]
----
<dependency>
    <groupId>io.hypersistence</groupId>
    <artifactId>hypersistence-utils-hibernate-73</artifactId>
    <version>3.15.4</version>
</dependency>
----

[NOTE]
====
The `-73` line targets Hibernate 7.3 *and Jackson 3* (`tools.jackson.*`), which
is the Jackson that Spring Boot 4 ships by default. Spring Boot 4.0.0 itself
manages Hibernate 7.1, so every example project overrides
`<hibernate.version>7.3.6.Final</hibernate.version>` — a property Boot's parent
POM honours.
====

The examples all model the bookstore's discussion forum: a `Post` is a thread, a
`PostComment` is a reply, `PostDetails` carries the thread's metadata, and a
`Tag` labels it. One rule holds across every project in this chapter: **`Post` is
the aggregate root.** A `PostDetails` or a `PostComment` never exists without the
`Post` it belongs to — if there is no thread, there is nothing to hold details
about.

== Identifiers that sort themselves: `@Tsid`

`AUTO_INCREMENT` identifiers leak your row count and force Hibernate to give up
JDBC batch inserts. `UUID` identifiers fix the leak and destroy your index
locality instead: random 128-bit values scatter writes across the whole B+Tree.

A TSID — Time-Sorted Identifier — is the compromise that costs nothing. It packs
a millisecond timestamp and a random component into 64 bits, so it fits a
`bigint`, it is generated by the application rather than the database, and
consecutive values are *increasing*, which is exactly what a B+Tree wants.

the database, which is the property that matters most:

----
include::{sourcedir}/HibernateSpringBootTsidIdentifier/src/test/java/com/bookstore/forum/PostBatchInsertTest.java[tag=batching]

Thirty posts, **zero** `SELECT` statements and exactly **one** `INSERT` — a
single JDBC batch. With `GenerationType.IDENTITY` this test cannot be written at
all: Hibernate must execute each `INSERT` on its own to learn the generated key,
so batching is silently disabled. The loop at the end asserts the other half of
the bargain: the identifiers come out in ascending order.

== JSON columns, portably

Every database spells JSON differently: MySQL has `json`, PostgreSQL has `json`
and `jsonb`, Oracle and SQL Server store it in text columns. `JsonType` hides
that difference behind one annotation, and it will map a POJO, a raw
`JsonNode`, or a plain `Map`:

.Three JSON columns, three Java shapes
[source,java]
----
include::{sourcedir}/HibernateSpringBootJsonType/HibernateSpringBootJsonTypeAttributes/src/main/java/com/bookstore/forum/entity/Post.java[tag=json-columns]
----

Hibernate 7 has native JSON support of its own —
`@JdbcTypeCode(SqlTypes.JSON)` — and for a plain document column you should
prefer it: it is dialect-aware, needs no `columnDefinition`, and works on all
four major databases. The repository carries both variants side by side so you
can compare them.

`JsonType` still earns its place for two reasons:

Content-level dirty checking:: `JsonType` compares the *parsed* JSON when
deciding whether a column changed. The native mapping compares object
identity, so a POJO without `equals` triggers an `UPDATE` on every flush — even
in a read-only transaction that touched nothing.
Reach:: `JsonNode`, `String` and older Hibernate versions are all supported.

=== A custom `ObjectMapper`

Both mappings let you supply your own Jackson configuration, through different
properties:

[source,properties]
----
# Hypersistence Utils
spring.jpa.properties.hypersistence.utils.jackson.object.mapper=\
  com.bookstore.forum.config.SnakeCaseObjectMapperSupplier

# Native Hibernate
spring.jpa.properties.hibernate.type.json_format_mapper=\
  com.bookstore.forum.config.SnakeCaseJsonFormatMapper
----

=== Polymorphic JSON, and a trap

Storing a heterogeneous list — text blocks, image blocks, code blocks — in one
JSON column needs a Jackson discriminator. The obvious mapping does not work
with `JsonType`, and the reason is worth understanding:

.The discriminator has to be a real bean property
[source,java]
----
include::{sourcedir}/HibernateSpringBootJsonType/HibernateSpringBootJsonTypePolymorphism/src/main/java/com/bookstore/forum/entity/PostContentBlock.java[lines=32..47]
----

`JsonType` serializes through `writeValueAsString(value)` *without* passing the
declared generic type, so Jackson's polymorphic type serializer never runs and a
plain `@JsonTypeInfo` discriminator is silently dropped on write. Exposing the
discriminator as a genuine `getType()` getter and declaring
`As.EXISTING_PROPERTY` makes it survive.

The native `@JdbcTypeCode(SqlTypes.JSON)` mapping does *not* have this problem —
it passes the resolved generic type — but the mapping above works for both, so
it is the one to reach for.

== Array columns

PostgreSQL arrays are a genuine column type, not a serialization trick, and
Hypersistence Utils maps every useful shape of them: `String[]`, `int[]`,
`List<String>`, `LocalDate[]`, `UUID[]`, enum arrays and even multidimensional
arrays.

.The two array types Hibernate cannot map natively
[source,java]
----
include::{sourcedir}/HibernateSpringBootArrayTypes/HibernateSpringBootHypersistenceArrayTypes/src/main/java/com/bookstore/forum/entity/Post.java[tag=enum-array]
----

For simple arrays, Hibernate 7 needs no annotation at all — declare
`private String[] keywords;` and you get a `text array` column. The two mappings
above are where that stops being true:

* An enum array maps natively to `smallint array` (ordinals), *not* to a
  PostgreSQL `post_status[]`. `EnumArrayType` with the `SQL_ARRAY_TYPE`
  parameter gives you the real enum array type.
* A multidimensional array is rejected outright: the `EntityManagerFactory`
  fails to start with _"Nested arrays (with the exception of `byte[][]`) are not
  supported"_. `MultiDimensionalArrayType` maps it.

[NOTE]
====
The PostgreSQL enum type has to exist before Hibernate generates the table. Put
`CREATE TYPE` in `schema.sql` and leave `spring.jpa.defer-datasource-initialization`
at its default of `false` — that ordering runs the script *before* the
`EntityManagerFactory` is built.
====

=== What Hibernate 7 now maps by itself

Not every one of these types still needs the library, and it is worth knowing
which. The repository has a second module with the *native* equivalents, so the
comparison is tested rather than asserted:

[cols="1,3", options="header"]
|===
| PostgreSQL type | Hibernate 7.3 support

| `interval`
| **Native.** `@JdbcTypeCode(SqlTypes.INTERVAL_SECOND)` on a `Duration` produces
  `interval second(6)`.

| `inet`
| **Native.** A plain `java.net.InetAddress` field maps to `inet` with no
  annotation at all.

| `citext`
| **Half.** `columnDefinition = "citext"` gives you a case-insensitive *unique
  constraint*, but Hibernate binds query parameters as `varchar`, so a derived
  query is still case-**sensitive** unless you write `CAST(:name AS citext)`.

| `tsrange`
| Hypersistence Utils only (`PostgreSQLRangeType`).

| `hstore`
| Hypersistence Utils only (`PostgreSQLHStoreType`).
|===

[WARNING]
====
When querying an `hstore` column, use the function form `exist(attributes, :key)`
rather than the `?` operator — JDBC parses `?` as a bind placeholder and the
statement never reaches PostgreSQL intact.
====

== A better repository contract: `BaseJpaRepository`

Spring Data's `JpaRepository.save()` is one method doing two jobs. Given a
detached entity it calls `merge()`, and `merge()` must first `SELECT` the row to
build a managed copy — a read you did not ask for and do not need, because you
already know what the row should look like.

`BaseJpaRepository` splits the method in two. Wire it in once:

[source,java]
----
@SpringBootApplication
@EnableJpaRepositories(repositoryBaseClass = BaseJpaRepositoryImpl.class)
public class MainApplication { /* ... */ }
----

.The repository interface
[source,java]
----
include::{sourcedir}/HibernateSpringBootBaseJpaRepository/src/main/java/com/bookstore/forum/repository/PostRepository.java[tag=repository]
----

Notice what is *missing*: there is no `save()` and no `findAll()`. That is the
whole point — you are made to choose between `persist` (this row is new) and
`update` (this row exists), and the choice is enforced by the compiler.

.The explicit contract: a direct UPDATE
[source,java]

include::{sourcedir}/HibernateSpringBootBaseJpaRepository/src/test/java/com/bookstore/forum/BaseJpaRepositoryTest.java[tag=base-update]

`update()` delegates to Hibernate's `StatelessSession`, which writes the row
without a persistence context and therefore without reading it first.

[IMPORTANT]
====
`updateAll()` removes the N `SELECT`s but it does **not** batch the `UPDATE`s.
`StatelessSessionImpl` sets the JDBC batch size to `0` for a single `update()`
call — a non-zero batch size would force write-behind, which is precisely what a
stateless session refuses to do. Three detached entities produce three
`UPDATE` statements. Compare that with `saveAll()`, which produces three
`SELECT`s *and* one batched `UPDATE`: you are trading reads for roundtrips, so
measure which one your workload actually cares about.
====

== Retrying what is worth retrying

Some failures are worth a second attempt and some are not. A lock acquisition
timeout is transient: whoever held the lock has since committed, and running the
transaction again will probably succeed. An optimistic locking failure is not
transient in the same sense — retrying it silently overwrites somebody else's
concurrent decision, which is the exact outcome the version column existed to
prevent.

`@Retry` is generic; you decide which exceptions qualify:

.Retrying only recoverable failures
[source,java]
----
include::{sourcedir}/HibernateSpringBootRetry/src/main/java/com/bookstore/forum/service/RetryableForumService.java[tag=retry]
----

The structure matters as much as the annotation. `@Retry` sits on a method that
is **not** `@Transactional`, and that method delegates to a *different* bean
which is. Crossing the proxy boundary is what gives each attempt a fresh
transaction and a fresh read of the row; annotating one method with both would
retry inside the failed transaction, which cannot succeed.

The repository proves this with two threads and a `CountDownLatch`: one holds a
`PESSIMISTIC_WRITE` lock while the other times out, retries, and commits once the
lock is released. Without `@Retry` the same scenario fails outright.

`SQLExtractor`:: *What SQL will this JPQL become?* It translates without
executing, which makes it a fine thing to assert on in a test:
+

----
include::{sourcedir}/HibernateSpringBootQueryUtilities/src/test/java/com/bookstore/forum/QueryUtilitiesTest.java[tag=sql-extractor]
----

`QueryStackTraceLogger`:: *Which of my methods fired this query?* Give it your
base package and it logs the application frame responsible — the fastest way to
find the origin of a query you did not expect. It only logs at `DEBUG`.

`JfrQueryLogger`:: *What did production actually run?* It emits a JDK Flight
Recorder event per query, so the SQL turns up in a JFR recording alongside GC
and thread data.

The last two are both `StatementInspector` implementations, and Hibernate accepts
only one. A no-argument composite solves it without any programmatic
configuration:

[source,java]
----

----

  com.bookstore.forum.config.CompositeStatementInspector

----

Hibernate instantiates that class reflectively, which is why the no-argument
constructor matters.

== Sequences that stay usable by everyone

A sequence roundtrip per inserted row is fine for a single-row transaction and
ruinous for a bulk import. The usual answer is the `pooled` optimizer: declare
the sequence `INCREMENT BY 50` and let Hibernate hand out blocks from memory.

It works, and it quietly makes the sequence Hibernate's private property. Any
other client — an ETL job, a `psql` script, a trigger — that calls `nextval`
takes one identifier and burns forty-nine, because the pooling lives in the
application, not in the database.

`BatchSequenceGenerator` gets the same roundtrip reduction without that. It
batches the *fetch* instead of inflating the increment, using a recursive CTE
(on PostgreSQL, `generate_series`) to draw many values in one statement:

====
`@BatchSequence` is not gap-free in the absolute sense — the unused tail of a
fetch is lost, so size `fetchSize` to your import chunk. What it preserves is

that anything can consume.

`@SequenceOptimizer` is the companion annotation for when you do want an
optimizer, chosen explicitly rather than inferred from `allocationSize`.

== Summary

Hypersistence Utils is not one feature; it is a collection of answers to
questions Hibernate leaves open. The ones worth remembering:

* `@Tsid` gives you sortable, batch-friendly, non-guessable identifiers in 64 bits.
* `JsonType` and the array types map the columns your database really has —
  though Hibernate 7 has caught up on plain JSON, simple arrays, `interval` and
  `inet`, so check before reaching for the library.
* `BaseJpaRepository` replaces `save()` with `persist()` and `update()`, and the
  `SELECT` before every detached update disappears.
* `@Retry` retries what is transient; you choose the exception list, and the
  annotation goes on a non-transactional method that delegates to a
  transactional one.
* `SQLExtractor`, `QueryStackTraceLogger` and `JfrQueryLogger` answer *what*,
  *who* and *when* about your SQL.
* `BatchSequenceGenerator` speeds up imports without turning your sequence into
  something only Hibernate can use.

Every claim above is a test in this book's repository. If one of them ever stops
being true, that build fails — which is the subject of the next chapter.
